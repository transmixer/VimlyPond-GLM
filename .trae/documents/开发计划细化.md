# VimlyPond 开发计划细化

## 一、决策

**已确认方案**：方案 B - 引入 `MusicElement` 抽象层重构数据结构

**实施策略**：
1. 先提交现有代码到 master，打 v0.10.0 标签
2. 创建 `feature/data-structure-v2` 分支进行重构
3. 重构期间暂停所有新功能开发
4. 确保单元测试和集成测试全部自动化
5. 重构完成后，确保所有现有功能正常工作

---

## 二、实施步骤

### Step 1: 提交现有代码 (v0.10.0)

**目标**：将现有代码作为稳定版本提交到 master

**操作**：
```bash
# 1. 确保当前代码是最新
git status
git pull origin master

# 2. 运行测试确保全绿
bun run test
bun run lint

# 3. 更新 CHANGELOG.md (如有未记录的变更)

# 4. 提交
git add -A
git commit -m "feat: v0.10.0 - 完成核心编辑功能"

# 5. 打标签
git tag -a v0.10.0 -m "v0.10.0 - 核心编辑功能完成"

# 6. 推送到远程
git push origin master
git push origin master --tags
```

**验收标准**：
- `bun run test` 全绿
- `bun run lint` 无错误
- GitHub 上有 v0.10.0 标签

---

### Step 2: 创建重构分支

**操作**：
```bash
git checkout -b feature/data-structure-v2
```

---

### Step 3: 设计并实现 v2 数据结构

#### 3.1 创建新类型文件 `src/lib/vimlypond/types-v2.ts`

核心类型：Note, Rest, Tuplet, GraceNote, Barline, MusicElement, Voice, Measure, Staff, StaffGroup, Score

#### 3.2 创建新音乐逻辑文件 `src/lib/vimlypond/music-v2.ts`

- 时值计算
- 八度推断
- 调号处理
- LilyPond 导出

#### 3.3 创建新状态管理 `src/lib/vimlypond/store-v2.ts`

- 保持与 store.ts 相同的 API 接口
- 所有操作基于 v2 类型

#### 3.4 创建新渲染器 `src/lib/vimlypond/renderer-v2.ts`

- 基于 v2 类型渲染
- 支持所有新功能

---

### Step 4: 创建迁移脚本 `src/lib/vimlypond/migrate.ts`

将 v1 数据转换为 v2 格式，保留用户数据。

---

### Step 5: 测试

#### 5.1 单元测试

每个新模块必须有对应的测试文件：

```bash
src/lib/vimlypond/types-v2.test.ts
src/lib/vimlypond/music-v2.test.ts
src/lib/vimlypond/store-v2.test.ts
src/lib/vimlypond/renderer-v2.test.ts
src/lib/vimlypond/migrate.test.ts
```

#### 5.2 集成测试

```bash
bun run test:e2e
```

#### 5.3 验收标准

- 所有单元测试通过 (`bun run test`)
- 所有集成测试通过 (`bun run test:e2e`)
- Lint 检查通过 (`bun run lint`)

---

### Step 6: 集成和替换

1. 替换 `VimlypondEditor.tsx` 使用 v2 版本
2. 完整测试套件验证

## 三、重构期间注意事项

### 3.1 暂停事项
- 禁止开发新功能
- 禁止向 master 分支推送代码
- 禁止在 master 上直接修复问题

### 3.2 必须事项
- 每次提交必须附带测试
- 重构期间保持代码可运行
- 确保 `bun run test` 始终全绿
- **每个功能模块重构前后都要单独提交，确保可回溯**
- **AI 负责在必要时进行代码回溯操作**

### 3.3 风险缓解
- 完整测试覆盖
- 提供数据迁移脚本
- 分阶段验证
