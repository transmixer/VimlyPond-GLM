# Vimlypond 设计/编码原则

> 每次对话执行前都要牢记！

## 核心原则

1. **TDD（测试驱动开发）**
   - 先写测试，再写实现
   - 修复 bug 时先写测试复现问题

2. **语法尽量参考 LilyPond**
   - 音符命名：a-g
   - 时值：1, 2, 4, 8, 16
   - 升降号：is/es（或 +/- 作为快捷键）
   - 八度：' 和 ,

3. **按键绑定尽量参考 Vim**
   - 普通模式/插入模式分离
   - h/j/k/l 导航
   - i/a 进入插入模式
   - ESC 返回普通模式
   - u 撤销

4. **修改设计时发生键位冲突必须二次确认**
   - 不能直接覆盖已有键位
   - 必须向用户确认后才能修改

5. **已修复的 bug 重新出现时，优先检查所修改的内容**
   - 检查相关 CSS 是否被改动
   - 检查相关常量是否被修改
   - 检查坐标系/尺寸计算是否一致

6. **流式推拉设计原则**
   - 每个小节必须满时值（4拍）
   - 分割产生延音线，合并消除延音线
   - 合并延音线前检查是否会导致溢出
   - 推 `tieEnd` 音符时，如果没有对应的 `tieStart` 前驱，清除 `tieEnd` 标记

## 流式推拉设计详解

### 问题背景

当音符时值变化时，小节可能出现不满或溢出的情况。需要设计机制确保每个小节都满时值。

### 解决方案

**分割与合并**：
- 分割：当需要将音符的一部分移动到另一个小节时，分割音符并添加延音线
- 合并：当延音线两端的音符可以合并时，消除延音线并合并音符

**关键规则**：

| 场景 | 处理方式 |
|------|---------|
| 小节溢出 | 推完整音符到下一小节，或分割带延音线 |
| 小节不满 | 从下一小节拉完整音符，或分割带延音线 |
| 延音线合并 | 检查合并后是否溢出，溢出则不合并 |
| 推 tieEnd 音符 | 无对应 tieStart 时，清除 tieEnd 标记 |

**核心函数**：
- `splitNoteWithDuration(note, beats)` - 分割音符
- `mergeTiedNotes(first, second)` - 合并延音线音符
- `balanceMeasures(staff, startFrom)` - 平衡小节时值

### 延音线方向

分割后的延音线方向：
- 第一个音符：`tieStart = true`
- 第二个音符：`tieEnd = true`

```
[小节1] ... | 音符(tieStart) |
[小节2] 音符(tieEnd) | ...
```

### 边界情况处理

1. **分割后立即合并**：`tryMergeFromPrevious` 检测合并后是否溢出
2. **推延音线后半部分**：`pushElementToNext` 清除 `tieEnd` 标记
3. **最后小节不满**：可选填充休止符

## 常见陷阱

### CSS 与 JS 常量不同步

**案例**：光标位置偏移 4px

- `renderer.ts` 定义 `SVG_PADDING = 20`
- `VimlypondEditor.tsx` 使用 `p-4`（16px）
- 差异 4px 导致光标定位错误

**教训**：CSS 类（如 `p-4`、`p-5`）与 JS 常量必须保持一致，修改任意一方都要检查另一方。

### 延音线合并导致溢出

**案例**：分割后又被自动合并

1. 小节1不满，分割下一小节音符，产生延音线
2. 处理下一小节时，检测到延音线，尝试合并
3. 合并后小节1溢出

**教训**：`tryMergeFromPrevious` 必须检查合并后是否溢出，溢出则不合并。
