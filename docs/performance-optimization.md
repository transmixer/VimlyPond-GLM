# Vimlypond 性能优化备忘

## 当前状态

流式推拉（Stream Push/Pull）功能性能测试通过，典型使用场景无性能问题。

### 性能基准测试结果

| 规模 | 小节数 | 推拉耗时 | 评价 |
|-----|-------|---------|------|
| 短作品 | 100 | ~0.3ms | ✓ 流畅 |
| 奏鸣曲乐章 | 300 | ~0.2ms | ✓ 流畅 |
| 完整奏鸣曲 | 600 | ~0.1ms | ✓ 流畅 |
| 大型作品 | 1200 | ~0.1ms | ✓ 流畅 |

**结论**：计算层性能充足，60fps (16ms) 阈值下有 100x 以上余量。

---

## 待优化项

### 1. VexFlow 增量渲染

**现状**：每次修改重绘整个乐谱

**优化方向**：
- 只重绘变化的谱表/小节
- 利用 VexFlow 的 `context.clear()` 局部清除
- 或维护音符元素映射，只更新变化元素

**预期收益**：渲染时间从 O(n) 降至 O(变化量)

**优先级**：中（当用户反馈大规模乐谱卡顿时处理）

---

### 2. React 虚拟化

**现状**：所有小节都在 DOM 中渲染

**优化方向**：
- 使用 `react-window` 或类似库
- 只渲染可视区域的小节
- 滚动时动态加载/卸载

**预期收益**：DOM 节点数恒定，不受乐谱长度影响

**优先级**：低（1200小节下当前 DOM 仍可接受）

---

### 3. 历史记录优化

**现状**：每次操作存储完整乐谱 JSON（最多 50 步）

**优化方向**：
- 改用操作日志（Operation Log）
- 存储增量差异而非完整快照
- 或使用 Immer 的 patch 系统

**预期收益**：内存占用降低，撤销栈容量可增加

**优先级**：低（当前内存占用未成为瓶颈）

---

## 不需要优化项

### 流式推拉计算

- 数组操作 + 简单算术，JavaScript 引擎优化良好
- 即使 1200 小节连锁推拉也只需 ~0.1ms
- 无需进一步优化

---

## 触发优化审查的条件

1. 用户报告 500+ 小节乐谱输入卡顿
2. 设备性能较低（移动端、老旧设备）
3. 内存占用超过 100MB
4. 单次操作耗时超过 50ms
